# Architecture

ConfidPay combines Fhenix CoFHE for encrypted computation with ReineiraOS for confidential payment rails.

## System Overview

```
┌─────────────────────────────────────────────────────────────────────┐
│                         User (DAO Admin)                            │
└─────────────────────────────────────────────────────────────────────┘
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    Frontend (@cofhe/sdk/web)                        │
│  • Client-side encryption via FhenixClient                          │
│  • Transaction submission                                            │
│  • Encrypted result decryption                                       │
└─────────────────────────────────────────────────────────────────────┘
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    ConfidPay.sol (on-chain)                         │
│  • Stores euint128 (encryptedAmount, encryptedNextRelease, etc.)    │
│  • ACL permissions (allowThis, allow)                               │
│  • Vesting calculations on encrypted data                           │
│  • Triggers escrow creation via ReineiraOS                          │
└─────────────────────────────────────────────────────────────────────┘
                                   │
                     ┌─────────────┴─────────────┐
                     ▼                           ▼
┌───────────────────────────────┐   ┌───────────────────────────────────┐
│   CoFHE Coprocessor           │   │       ReineiraOS Layer            │
│   (off-chain FHE compute)     │   │   (confidential payment rails)    │
│                               │   │                                   │
│   • Encrypted arithmetic      │   │   • ConfidentialEscrow            │
│   • Time-based release logic  │   │   • Insurance coverage            │
│   • Vesting calculations      │   │   • IFHERC20 transfers            │
└───────────────────────────────┘   └───────────────────────────────────┘
```

## Fhenix CoFHE Layer

### Encrypted Types

Fhenix provides Solidity-compatible encrypted types:

| Type | Description | Use Case |
|------|-------------|----------|
| `euint128` | Encrypted 128-bit unsigned integer | Salary amounts, balances |
| `euint64` | Encrypted 64-bit unsigned integer | Timestamps |
| `ebool` | Encrypted boolean | Release conditions |
| `eaddress` | Encrypted address | Recipient wallets |

### Operations

```solidity
// Encrypted arithmetic
euint128 encryptedSum = FHE.add(encryptedA, encryptedB);
euint128 encryptedProduct = FHE.mul(encryptedAmount, encryptedRate);

// Comparison (runs on encrypted data!)
ebool canRelease = FHE.le(encryptedNextRelease, currentTime);

// Conditional logic - USE FHE.select(), NOT if()
euint128 releaseAmount = FHE.select(
    canRelease,
    FHE.div(FHE.mul(encryptedAmount, encryptedRate), FHE.asEuint128(100)),
    FHE.asEuint128(0)
);
```

### ACL (Access Control List)

Critical for security — controls who can decrypt:

```solidity
// Allow the contract itself to access
FHE.allowThis(encryptedAmount);

// Allow specific recipient to decrypt their own data
FHE.allow(encryptedAmount, employee);  // NOT allowSender!

// Allow authorized auditor
FHE.allow(encryptedAmount, auditorAddress);
```

<Warning>
**Important:** `FHE.allowSender()` grants access to the **transaction caller** (admin), NOT the employee. Always use `FHE.allow(value, employee)` for employee access.
</Warning>

<Warning>
Missing `FHE.allow()` or `FHE.allowThis()` calls can permanently lock funds. Always grant access to the contract and intended recipient.
</Warning>

## ReineiraOS Layer

ReineiraOS handles confidential payment flows through escrow infrastructure:

- **ConfidentialEscrow** — Hold funds until conditions are met
- **InsuranceManager** — Coverage on escrow amounts
- **IFHERC20** — Confidential USDC transfers

### Payment Flow

```
1. Admin → createPayroll() → Salary stored encrypted
2. Admin → createPaymentEscrow() → Escrow created
3. Admin → fundEscrow() → Funds escrow + insurance auto-attached
4. Employee → getMyPendingEscrows() → View pending escrows
5. Employee → redeem() via SDK → Employee gets tokens
```

## SDK Integration

The `@cofhe/sdk` provides client-side encryption:

```typescript
import { FhenixClient, InEuint128 } from '@cofhe/sdk/web';

const client = new FhenixClient();

// Encrypt sensitive data client-side
const encryptedSalary = await client.encrypt(salaryAmount);

// Write to contract
await write('createPayroll', [
  recipient,
  { publicAmount: encryptedSalary },  // InEuint128 format
  { publicAmount: encryptedNextRelease },
  { publicAmount: encryptedVestingRate },
]);

// Decrypt results (only for authorized parties)
const plaintext = await client.decrypt(encryptedResult);
```

<Warning>
**Note:** `@cofhe/react` is in development. Use `@cofhe/sdk/web` directly for frontend integration.
</Warning>

## Data Flow

1. **Encryption**: Admin encrypts payroll data client-side
2. **Submission**: Transaction submitted with encrypted inputs
3. **Storage**: Contract stores encrypted values (not readable on-chain)
4. **Computation**: CoFHE coprocessor performs calculations on encrypted data
5. **Escrow Creation**: ConfidPay creates escrow via ReineiraOS
6. **Funding**: Admin funds escrow with USDC
7. **Redemption**: Employee redeems via SDK

## Composability

Other contracts can interact with ConfidPay's encrypted balances:

```solidity
// Other contracts see only encrypted data
mapping(address => euint128) public encryptedBalances;

// Can integrate with DeFi protocols without exposing amounts
function lendToProtocol(uint256 protocolId) external {
    euint128 amount = encryptedBalances[msg.sender];
    // Operates on encrypted data!
}
```
